# FormCraft v1.5 — Self-Optimizing Forms

## What v1 shipped

- Form builder (Next.js): field editor, drag-and-drop reorder, logic branching, variable interpolation, theming
- Renderer (SolidJS): conversational one-at-a-time rendering, localStorage draft persistence, file upload
- Backend: Supabase Postgres, Auth, Storage
- REST API with API key auth
- Responses dashboard

v1 is a capable form builder. v1.5 makes those forms intelligent.

---

## v1.5 Value Proposition

After v1.5 ships, a form creator can:

1. Define multiple phrasings of the same question (variants)
2. Publish the form — traffic is automatically split across variants
3. Come back days later and see which variant is winning, with real metrics
4. Let the system gradually shift traffic toward the best-performing variant automatically

The form gets better on its own. The creator does nothing after setup.

**What the system will know how to do:**
- Track every session with device, referrer, and timing metadata
- Record every question interaction event (shown, answered, skipped, abandoned, backtracked)
- Assign variants deterministically per session (user always sees the same variant they were assigned)
- Score variants by a composite of completion rate, answer time, and abandonment penalty
- Rebalance traffic weights automatically, on a schedule, once a variant clears the minimum impression threshold
- Never kill a variant entirely — always keep an exploration floor
- Show per-variant analytics in the builder dashboard

## Architecture Changes

### Core concept addition: Question Groups + Variants

Today: a `field` in the `fields` table is the atomic unit.

v1.5: a `field` can optionally belong to a `question_group`. The group holds multiple `question_variants`. At runtime, the renderer asks the backend which variant to show for each group, and that assignment is locked for the session.

Fields without a group continue to work exactly as before. This is fully additive at the renderer protocol level.

## Database Changes

### New tables

**`question_groups`**
```sql
id uuid primary key
form_id uuid references forms(id) on delete cascade
label text                        -- internal name ("annual revenue question")
optimization_goal text not null default 'completion_rate'
                                  -- 'completion_rate' | 'answer_time' | 'composite'
strategy text not null default 'epsilon_greedy'
                                  -- 'epsilon_greedy' | 'equal' (equal = fixed A/B, no rebalancing)
created_at timestamptz
```

**`question_variants`**
```sql
id uuid primary key
question_group_id uuid references question_groups(id) on delete cascade
variant_label text               -- "A", "B", "C" or custom name
title text not null
description text
type text not null               -- same field types as today
config jsonb not null default '{}'
logic jsonb not null default '[]'
traffic_weight numeric not null default 1.0   -- normalized by engine; raw weight not %
is_active boolean not null default true
created_at timestamptz
```

**`form_sessions`**
```sql
id uuid primary key default gen_random_uuid()
form_id uuid references forms(id) on delete cascade
started_at timestamptz not null default now()
completed_at timestamptz
device_type text                 -- 'mobile' | 'desktop' | 'tablet'
referrer text
metadata jsonb not null default '{}'
```

**`variant_assignments`**
```sql
session_id uuid references form_sessions(id) on delete cascade
question_group_id uuid references question_groups(id) on delete cascade
variant_id uuid references question_variants(id)
assigned_at timestamptz not null default now()
primary key (session_id, question_group_id)
```

**`question_events`**
```sql
id uuid primary key default gen_random_uuid()
session_id uuid references form_sessions(id) on delete cascade
variant_id uuid references question_variants(id) on delete cascade
                                  -- null for events on ungrouped fields
field_id uuid references fields(id) on delete cascade
                                  -- null for events on variants
event_type text not null          -- 'shown' | 'answered' | 'skipped' | 'abandoned'
                                  --  | 'backtracked' | 'validation_error'
duration_ms integer               -- time from 'shown' to this event
metadata jsonb not null default '{}'
created_at timestamptz not null default now()
```

**`optimization_runs`**
```sql
id uuid primary key default gen_random_uuid()
question_group_id uuid references question_groups(id) on delete cascade
ran_at timestamptz not null default now()
variant_scores jsonb not null     -- snapshot: [{variant_id, impressions, score, new_weight}]
action text                       -- 'rebalanced' | 'skipped_low_data' | 'skipped_no_change'
notes text
```

### Changes to existing tables

**`fields`** — add nullable foreign key:
```sql
alter table fields add column question_group_id uuid references question_groups(id) on delete set null;
```

A field with `question_group_id` set is the "canonical" field — it remains for backward compat with existing logic and API. At runtime, the renderer uses the assigned variant instead.

**`responses`** — add session linkage:
```sql
alter table responses add column session_id uuid references form_sessions(id) on delete set null;
```

This links a completed response back to its session so downstream conversion can be tracked later.

## New API Endpoints

### Session + Variant Assignment

```
POST /api/v1/forms/:id/sessions
```
Creates a form session. Returns `session_id`. Called by renderer on form open.

Body: `{ device_type?, referrer?, metadata? }`
Response: `{ session_id }`

```
POST /api/v1/forms/:id/sessions/:sessionId/assign
```
Given a session, returns variant assignments for all active question groups in this form.

Response:
```json
{
  "assignments": [
    {
      "question_group_id": "...",
      "variant_id": "...",
      "variant": { "title": "...", "type": "...", "config": {}, "logic": [] }
    }
  ]
}
```

Assignment logic (server-side):
- If assignment already exists for this session+group → return it (idempotent)
- Otherwise: weighted random selection from active variants using current `traffic_weight` values
- Write to `variant_assignments`

### Event Tracking

```
POST /api/v1/sessions/:sessionId/events
```

Body:
```json
{
  "events": [
    {
      "variant_id": "...",       // or field_id for ungrouped fields
      "field_id": "...",
      "event_type": "answered",
      "duration_ms": 4200,
      "metadata": {}
    }
  ]
}
```

Batched. Renderer buffers events and flushes every 5s or on page unload.

### Question Groups (Builder API)

```
GET    /api/v1/forms/:id/groups
POST   /api/v1/forms/:id/groups
PATCH  /api/v1/forms/:id/groups/:groupId
DELETE /api/v1/forms/:id/groups/:groupId

GET    /api/v1/forms/:id/groups/:groupId/variants
POST   /api/v1/forms/:id/groups/:groupId/variants
PATCH  /api/v1/forms/:id/groups/:groupId/variants/:variantId
DELETE /api/v1/forms/:id/groups/:groupId/variants/:variantId
```

### Analytics

```
GET /api/v1/forms/:id/groups/:groupId/analytics
```

Response per variant:
```json
{
  "variant_id": "...",
  "label": "A",
  "impressions": 340,
  "answer_rate": 0.82,
  "completion_rate": 0.74,
  "avg_answer_time_ms": 6200,
  "abandonment_rate": 0.18,
  "traffic_weight": 0.6,
  "is_winner": true
}
```

## Renderer Changes

### Session lifecycle

On form open:
1. Check `localStorage` for existing `session_id` for this form
2. If none → `POST /api/v1/forms/:id/sessions` → store `session_id` in localStorage
3. `POST /api/v1/forms/:id/sessions/:sessionId/assign` → get variant assignments
4. Merge variant content into field list: for each field with a `question_group_id`, replace its title/type/config/logic with the assigned variant's content

### Event emission

Events are buffered in memory. Flush triggers:
- Every 5 seconds (if buffer non-empty)
- On `visibilitychange` to hidden (tab switch / close) — use `navigator.sendBeacon` for reliability
- On form completion

Event construction:
- `shown` — emitted when a question enters the viewport / becomes active slide
- `answered` — emitted on confirmed answer (Next pressed or auto-advance)
- `skipped` — emitted when user skips an optional question
- `backtracked` — emitted when user presses Back from a question
- `abandoned` — inferred server-side from sessions with no `completed_at` after TTL; not emitted by renderer directly

Duration tracking: record `shown_at` timestamp per question; compute `duration_ms` at answer/skip/back time.

### Session consistency

- `session_id` stored in `localStorage` keyed by `fc_session_{form_id}`
- Separate from draft key `fc_draft_{form_id}`
- On form restart (existing restart button): new session is created, old session_id cleared

## Builder UI Changes

### Question Group editor

In the field editor panel, a new "Variants" section appears when a field belongs to a group (or when the user clicks "Add Variant").

Flow:
- User selects a field → clicks "Turn into experiment"
- A question group is created for that field; the field becomes Variant A
- User can add Variant B, C etc. — each has its own title, type, config, logic
- Each variant shows its current traffic weight (read-only, managed by engine)
- User can manually force-reset to equal weights

Visual indicator on field card in the builder list: a small "A/B" badge when a field has active variants.

### Analytics tab (per form)

New tab in the form detail view alongside "Builder" and "Responses": **"Experiments"**

Shows:
- List of all question groups for this form
- Per group: table of variants with impressions, answer rate, avg time, traffic share, winning badge
- Status: "Collecting data", "Optimizing", "Winner found" based on impression count and weight delta
- Last optimization run timestamp

No charts in v1.5. Table view only.

## Optimization Engine

### Where it runs

Supabase scheduled function (pg_cron or Edge Function triggered by cron). Runs every 15 minutes.

### Logic

For each active `question_group`:

1. Count impressions per variant from `question_events` where `event_type = 'shown'`
2. Skip if any variant has fewer than 100 impressions (write `optimization_runs` row with `action = 'skipped_low_data'`)
3. Compute score per variant:
   ```
   answer_rate      = answered_events / shown_events
   avg_time_penalty = clamp(avg_duration_ms / 30000, 0, 1)  -- penalize if >30s avg
   score = answer_rate * (1 - 0.3 * avg_time_penalty)
   ```
4. Normalize scores to weights, applying exploration floor:
   ```
   exploration_floor = 0.05                          -- each variant gets at least 5%
   remaining = 1 - (n_variants * exploration_floor)
   weight[i] = exploration_floor + (score[i] / sum_scores) * remaining
   ```
5. Update `traffic_weight` on `question_variants`
6. Write `optimization_runs` row with `action = 'rebalanced'` and snapshot

### Statistical guard

If all scores are within 0.03 of each other → `action = 'skipped_no_change'`. Do not update weights. Prevents thrashing when variants are statistically indistinguishable.

## Change Categories

### Breaking changes

- **Renderer form load protocol changes.** The renderer currently does a direct Supabase query from the client. v1.5 introduces a session creation + assignment step before the form can be rendered. This is a required flow change — no fallback to old protocol.
- **`responses` table gains nullable `session_id` column.** Existing responses have `null`. Analytics that join responses to sessions will miss historical data. Acceptable for v1.5.
- **REST API consumers: field objects may now have a `question_group_id` field.** Not breaking in the strict sense, but clients that mirror the schema need to handle the new field.

### Minor changes (updates to existing behavior)

- Field cards in the builder show an "A/B" badge when they belong to a group
- Form restart (already exists) now also creates a new session
- Draft persistence in localStorage is unchanged; session persistence is additive (new key)
- `POST /api/v1/forms/:id/responses` (external API) should accept an optional `session_id` in the body to link the response to its session. If omitted, response is saved unlinked — no breakage.
- RLS policies on new tables: `form_sessions` and `question_events` must be writable by anonymous (public) users since the renderer is unauthenticated. `question_groups`, `question_variants`, `variant_assignments`, `optimization_runs` are owner-only.

### Feature additions (net new)

- Question groups — data model + builder UI
- Question variants — create/edit/deactivate per group
- Session tracking — `form_sessions` created per form open
- Variant assignment — server-side weighted random, deterministic within session
- Event tracking — renderer emits batched events, beacon on unload
- Optimization engine — scheduled job scoring and reweighting variants
- Exploration floor enforcement — variants never fully killed
- Experiments tab in builder — per-group analytics table
- New API surface: sessions, assign, events, groups, variants, analytics endpoints

## What v1.5 does NOT include

These are explicitly out of scope:

- AI-generated variants (Phase 3)
- Contextual bandits (segment by device/geo/referrer) — data is collected but not used for routing
- Thompson Sampling or Bayesian optimization — epsilon-greedy only
- Downstream conversion events beyond form completion
- Time-series charts in the analytics UI
- Email notifications when a winner is found
- Multi-variate testing across multiple groups simultaneously (groups optimize independently)

## Edge Cases and Constraints

**Variant deactivation mid-session:** If a variant is deactivated while a user is mid-session, their assignment is already locked in `variant_assignments`. The renderer re-fetches assignments only on session start, not mid-form. The deactivated variant continues to serve for in-progress sessions. New sessions will not be assigned to it.

**All variants deactivated:** If all variants in a group are deactivated, fall back to the canonical field (the one in the `fields` table). Do not 500.

**Exploration floor with 2 variants:** Floor is 5% each = 10% reserved. Winner gets 90%. Acceptable.

**Exploration floor with 6+ variants:** Floor * n_variants could exceed 1.0. Cap: if `n_variants * floor > 0.5`, reduce floor to `0.5 / n_variants`. This prevents pathological cases.

**Race condition on assignment:** Two simultaneous requests for the same session+group. DB unique constraint on `(session_id, question_group_id)` in `variant_assignments` handles this — first write wins, second gets a conflict error and should re-read.

**sendBeacon failure on unload:** Best-effort only. Incomplete sessions are expected. Abandonment is inferred server-side, not from a client event.

**Form with no groups:** Renderer flow adds session creation step even for forms without any question groups. The assign endpoint returns an empty `assignments` array. Slight latency increase (~1 extra round trip) for all forms. Acceptable.

**Minimum impression threshold and new variants:** Adding a new Variant C to an existing experiment that already has 1000 impressions on A and B — C starts at 0 impressions. Engine skips optimization for the whole group until C clears 100 impressions. This means A/B weights freeze temporarily. Document this behavior; it is intentional to avoid starving the new variant.

**Clock skew and duration_ms:** `duration_ms` is computed client-side. Treat it as approximate. Do not use it for billing or SLA.

## Post-v1.5 System State

A developer using FormCraft after v1.5 ships:

- Builds a form exactly as before
- Optionally marks any question as an experiment, adds 2–3 phrasings
- Publishes — no extra steps
- Returns a week later to the Experiments tab
- Sees which variant is winning and by how much
- The form is already routing 60–70% of traffic to the winner automatically
- Can add a new variant at any time; the engine will start testing it

The product is still a form builder. But every published form is also a running experiment.
